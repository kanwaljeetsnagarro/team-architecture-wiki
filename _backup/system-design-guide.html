System Design Interview Preparation Guide for Architect Roles

Framework, building blocks, five solved design problems, and real-project connection strategy.

Part 1 - The 7-Step Framework

System design interviews test structured thinking and trade-off clarity, not memorized diagrams.

StepGoalTypical time
1. Clarify requirementsFunctional + non-functional scope questions3-5 min
2. Define in/out scopeState what will and will not be covered1-2 min
3. Capacity estimationRPS, data size, growth, peak assumptions3-5 min
4. High-level designMajor boxes and interactions10-15 min
5. Deep diveCritical components and failure handling10-15 min
6. Bottlenecks + trade-offsName what breaks first and chosen compromise5 min
7. Wrap-upSummary and what you'd do next2 min

Questions that signal seniority

  • Scale: users, peak RPS, read vs write ratio
  • Consistency: strong vs eventual needs
  • Latency: user-facing SLA expectations
  • Failure tolerance: acceptable degradation and data loss tolerance
  • Constraints: legacy integrations and technology bounds

CAP theorem

Under network partition, distributed systems choose between consistency and availability (partition tolerance is non-negotiable in real systems).

ChoiceMeaning
CPPrefer consistent state, may reject requests during partition
APPrefer availability, may serve temporarily stale data

Problem 1 - Design a URL Shortener

Clarify requirements

  • Shorten long URL to short code
  • Redirect short URL to original
  • Optional alias, expiry, click analytics

Capacity assumptions

100M new URLs/month
10B redirects/month
Read:Write ~100:1
~3TB storage over 5 years

High-level design

Client -> LB -> API Gateway
Write path: shorten service -> code generation -> DB -> cache
Read path: redirect service -> cache hit/miss -> DB -> 302

Short code generation trade-off

ApproachProsCons
Hash + truncateSimple deterministicCollision handling required
Counter + Base62No collisions, deterministicNeeds reliable distributed ID strategy
Q: Custom alias without race condition?

Enforce DB unique constraint and handle unique-violation conflict (409). Do not rely on check-then-write only.

Problem 2 - Design a Distributed Rate Limiter

Algorithm comparison

AlgorithmStrengthWeakness
Fixed WindowSimpleBoundary burst issue
Sliding LogAccurateHigh memory usage
Sliding CounterBalanced accuracy/memorySlight approximation
Token BucketAllows controlled burstsMore complex implementation
Leaky BucketSmooth output rateLess burst-friendly for clients
Interview recommendation: token bucket for client-facing APIs + sliding-window counter for backend shaping. 
Gateway instances -> shared Redis state
Use atomic Lua script for check-and-decrement
Avoid per-instance memory counters (incorrect at scale)
Q: API key limit + unauthenticated global limit together?

Apply layered checks with separate keys: rate_limit:ip and rate_limit:api_key. Both must pass.

Problem 3 - Multi-Tenant SaaS API Gateway

Requirements highlights

  • Tenant-aware routing and isolation
  • Auth and per-tier rate limits
  • Low gateway overhead latency
  • High availability for all tenants

Tenant resolution options

ApproachDetailTrade-off
Token claimTenant ID embedded in JWT/OAuthFastest request-time lookup
Header-basedTenant-ID headerNeeds additional verification
Subdomain-basedtenantX.api.company.comGood for portal-style patterns

Isolation strategies

StrategyStrengthCost/Risk
Dedicated infrastructureStrongest isolationHighest cost
Shared with row-level isolationCost-efficientData leak risk if filter bug
HybridTier-based balanceMore operational complexity
Q: One slow tenant backend affecting all?

Use per-tenant circuit breaker and bulkhead resource pools so unhealthy tenant calls cannot starve shared gateway capacity.

Problem 4 - Notification System (Email, SMS, Push)

High-level asynchronous design

Producer service publishes NotificationRequested event
Notification orchestrator:
 - user preference check
 - template render
 - channel decision
Publish to channel queues:
 - email queue
 - sms queue
 - push queue
Workers call provider APIs and write delivery status logs

Why separate queues per channel?

  • Different provider rate limits
  • Failure isolation across channels
  • Independent retry policy tuning

Idempotency requirement

Caller-generated idempotency keys prevent duplicate sends across retries and restarts.

Q: How to prioritize critical notifications?

Use priority queues and dedicated worker pools so critical transactional alerts are not delayed by bulk marketing load.

Problem 5 - Legacy Monolith Modernization Strategy

Strategy options

StrategyTrade-off
Big bang rewriteCleanest end-state, highest risk and delayed value
Strangler FigLower risk incremental value, longer transition
Parallel runHighest confidence, highest temporary cost

Incremental strangler phases

Phase 1: Establish seam (facade/ACL)
Phase 2: Extract low-coupling capabilities
Phase 3: Add modern integration surface around core
Phase 4: Reassess what should remain legacy vs extracted

When not to decompose further

SignalWhy stop
Deep shared DB couplingDistributed transaction complexity may outweigh benefit
High per-customer customizationMigration cost multiplies significantly
Stable low-change coreIndependent deployability benefit may be marginal
Q: How to measure modernization success?

Track leading indicators: deployment frequency, incident rate, developer velocity, latency/error rate, and user-impact metrics.
