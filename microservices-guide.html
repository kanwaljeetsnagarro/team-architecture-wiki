Microservices Architecture Team Knowledge Guide

Fundamentals, design process, communication, data patterns, resilience, testing, additions, and interview Q&A.

Section 1 - What Are Microservices?

Microservices architecture builds an application as a set of small independently deployable services. Each service owns one business capability, process boundary, and usually one data boundary.

ArchitectureWhat it isWhen it fits
MonolithSingle deployable codebaseSmall team, early product, MVP
Modular MonolithSingle deployable with strict module boundariesGrowing product without distributed overhead
MicroservicesMultiple independent deployables and data ownersMultiple teams and independent scaling needs
A well-designed monolith is better than poorly-operated microservices. Use microservices only when independence needs justify complexity.

6 core principles

  • Single responsibility
  • Independent deployability
  • Independent scalability
  • Database per service
  • Loose coupling
  • High cohesion
Fundamentals Q&A

Are microservices always better? No. They increase operational complexity and only pay off at certain scale/team structures.

How to choose service boundaries? Start from business bounded contexts using domain language, not technology layers.

What is a bounded context? Domain boundary where a specific model and vocabulary stay consistent.

Architecture design Q&A

BFF pattern? Separate backend layer per client type (mobile/web/partner), each optimized for that client's needs.

Shared functionality across services? Prefer duplication for small utility logic; shared library for non-domain utilities; shared service only when centralization is justified.

Communication Q&A

REST vs queue? REST for immediate dependency; queue for background decoupled work.

Eventual consistency acceptable when? When slight state delay does not harm user/business guarantees.

Idempotency why critical? Message redelivery happens; handlers must avoid duplicate side effects.

Section 4 - Data Management Patterns

When transactions span services, single ACID transaction no longer applies.

Problem:
Create order succeeds
Charge payment fails
System inconsistent

Saga solution:
Step1 create order
Step2 charge payment fails
Compensate: cancel order

Saga pattern

StepForward actionCompensation
1Create orderCancel order
2Charge customerRefund customer
3Reserve inventoryRelease reservation
4Generate invoiceVoid invoice
5Send confirmationBest-effort
Compensation is not DB rollback. It is a new business action that reverses prior committed effects.

Outbox pattern

In one DB transaction:
 - Save business record
 - Save event in outbox table
Background worker publishes outbox events later
No dual-write data loss window

CQRS

SideCharacteristics
Command side (write)Business rules and consistency focus
Query side (read)Read-optimized denormalized views
Data management Q&A

Saga use case? Multi-service business transaction requiring consistency and compensations.

Outbox why needed? Guarantees event persistence with data write in one transaction boundary.

Rollback vs compensation? Rollback undoes uncommitted local changes; compensation is new action after commit.

Section 5 - Resilience and Reliability

Failure typePattern
Transient network blipRetry with exponential backoff
Slow downstreamTimeout
Sustained failuresCircuit breaker
Resource starvationBulkhead
Poison messageDead letter queue
Pod crashLiveness probe restart
Not ready yetReadiness probe remove from LB
Traffic spikesHorizontal autoscaling
Expensive repeated lookupsCache-Aside
Circuit states:
Closed -> normal operation
Open -> fail-fast, no downstream calls
Half-open -> trial request for recovery test

Retry best practices

  • Exponential backoff + jitter
  • Max retry cap
  • Retry only transient errors
  • Retry only idempotent operations
Resilience Q&A

Retry vs circuit breaker? Retry handles transient blips; circuit breaker handles sustained unhealthy downstream behavior.

DLQ handling? Move failed messages aside after max retries, investigate root cause, then replay safely.

Zero downtime deployment? Rolling/canary deploys + readiness checks + backward-compatible contracts + feature flags.

Testing and observability Q&A

How to test without deploying everything? Layered strategy: many unit tests, fewer integration tests, targeted contract tests, minimal E2E.

Why structured logging? Machine-queryable fields enable fast distributed debugging with correlation IDs.

Must-have metrics? request rate, error rate, p50/p95/p99 latency, resource metrics, business metrics, and dependency metrics.

Section 7 - General Interview Q&A and Decision Guides

How to handle breaking API changes?

Version APIs, keep old version running, deprecate clearly, and preserve backward compatibility discipline.

Service discovery why needed?

In dynamic environments addresses change; service discovery provides stable lookup names (e.g., Kubernetes DNS).

Security model in microservices?

Edge auth at gateway, mTLS internally, least-privilege access, and secret store for credentials.

Migration challenges from monolith?

Operational complexity, distributed failure modes, consistency management, testing complexity, and team ownership shifts.

Types Q&A

Aggregator vs BFF? Aggregator is general data combiner; BFF is dedicated per client type and team.

When sync vs async? Sync when immediate next-step dependence exists; async for background and decoupled operations.

Risk of long synchronous chains? Latency and fragility compound with each hop.

Addition B - WhatsApp vs Phone Call Analogy

Group chat analogy for publish-subscribe

  • Direct message = point-to-point queue
  • Group message = one publisher, many subscribers
  • Adding new subscriber does not require publisher change

Analogy Q&A

How to explain sync vs async to non-technical person? Phone call requires immediate presence; WhatsApp decouples time.

Why async is more resilient? Work queues when consumer is down and resumes on recovery instead of immediate caller failure.
