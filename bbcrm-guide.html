Section 1 - BBCRM Functional Overview

Blackbaud CRM (BBCRM) is an enterprise CRM platform for mission-driven organizations including non-profits, universities, hospitals, museums, and foundations. It supports donor lifecycle management, campaigns, events, memberships, and financial integration.

Business DomainWhat it manages
Constituent ManagementDonors, alumni, volunteers, organizations
Revenue & DonationsGifts, pledges, recurring donations, soft credits, matching gifts
MembershipTiers, renewals, expiry, benefits tracking
EventsRegistration, attendance, event revenue
CampaignsFundraising campaigns and attribution
Batch ProcessingBulk import and transaction processing
Finance IntegrationGeneral ledger, revenue recognition, reconciliation
Reporting & AnalyticsAd-hoc queries, KPI dashboards, executive reporting

Deployment model

CharacteristicDetail
Dedicated environmentsEach customer has isolated DB, app server, and config
Release modelQuarterly service packs; customers choose upgrade timing
CustomizationCustom fields, specs, and logic per customer
IsolationOne customer's issue does not impact another
Functional Q&A

What organizations use BBCRM? Non-profits, universities, hospitals, foundations, and cultural institutions focused on fundraising and constituent engagement.

Constituent vs Donor? Constituent is any entity in system. Donor is a constituent who has given financially.

What is a Service Pack? Quarterly packaged release with features and fixes, deployed on customer schedule.

Technical Q&A

Metadata-driven architecture? Engine behavior is driven by XML specs at runtime. You can add forms and workflows via configuration metadata.

Why stored procedures? Historical enterprise design choice for transaction control and DB optimization; trade-off is split logic and migration complexity.

What is BizOps? Typed business service layer on top of Infinity for operations like constituent creation and donation processing.

Translation Layer Q&A

One-sentence definition: Cloud-native bridge converting REST/JSON requests into SOAP/XML calls and responses.

Is it a microservice? Deployed like one, but it is an integration service rather than business-domain owner.

How does operation routing work? endpoint -> operation_id -> configuration lookup -> mapping and transformation rules.

How does PATCH edit work? GET existing record, merge incoming partial fields, submit complete form expected by legacy model.

Security Q&A

Authentication vs authorization? Authentication verifies identity; authorization checks permission scope after identity is confirmed.

Common BBCRM gotcha? SSO login may pass, but BBCRM access still fails if app-user record/mapping is missing.

Why mTLS? Both client and server authenticate each other, reducing impersonation risk in internal network traffic.

BBCRM release stages

  1. File drop (artifacts + XML specs)
  2. Database setup (restore, encryption, migrations)
  3. IIS setup (app pools, sites, bindings)
  4. Web API workflow (SOAP endpoints + validation)

Translation layer build and release sequence

  • Build: restore, compile, test, coverage, Docker image, push image, publish OpenAPI/artifacts
  • Release zones: test -> US -> Canada -> Europe -> Australia -> post-deploy APIM update

Canary rollout model

Deploy new version beside current
Start at 0% traffic
Health checks pass
Shift traffic: 10% -> 25% -> 50% -> 100%
If any issue: instantly remove new version and keep old traffic
CI/CD Q&A

Main model difference? BBCRM is classic infra-owned quarterly release; translation layer is modern engineering-owned frequent delivery.

Why canary? Limits blast radius and enables fast rollback with minimal user impact.

Why multi-stage Docker build? Smaller and safer runtime image by separating build and runtime dependencies.

Section 6 - Architecture Patterns and Design Decisions

Why not rewrite BBCRM fully into microservices?

ObstacleImpact
Shared database + cross-domain stored proceduresHard to decompose into true database-per-service boundaries
Deeply coupled ACID operationsWould require sagas and compensations across many domains
Metadata runtime couplingNo clean decomposition seams without large runtime re-platform
High customer customization volumeMigration complexity multiplies across installations
Can extractCannot easily extract
Notifications, reporting, document generation, integration adaptersConstituent core, gift processing, cross-domain security, broad batch operations
Strategic answer: modernize at integration boundaries using Strangler Fig and Anti-Corruption Layer; keep proven business engine stable.

Patterns actively used

  • Anti-Corruption Layer
  • Strangler Fig
  • Config-driven behavior
  • Cache-Aside
  • Canary deployment
  • Correlation ID tracing
Architecture Q&A

What is Anti-Corruption Layer? Translator boundary that prevents legacy model leakage into modern service contracts.

What is Strangler Fig? Incremental modernization approach that grows new capabilities around legacy core.

Why correlation ID? Enables full end-to-end trace lookup for one request across all services and logs.

Section 7 - General Interview Q&A

How does request flow end-to-end?

Client -> APIM -> Translation (context + config + mapping) -> BBCRM AppFx -> Infinity + DB -> translated JSON response with correlation trace.

Synchronous vs asynchronous communication?

Synchronous waits for immediate response; asynchronous publishes work and continues. APIs often sync; notifications/analytics often async.

Horizontal vs vertical scaling?

Vertical adds resources to one node; horizontal adds instances. Translation layer scales horizontally due to stateless design.

Dependency injection value?

Externalizes dependencies, improves testability, and supports implementation swapping with minimal code change.

How to debug distributed production issues?

Start with correlation ID and inspect layers systematically: auth, context resolution, translation, downstream business operation.

How to add a new API endpoint?

Design consumer contract, map operation configuration, implement endpoint, test translation logic and end-to-end behavior.

How to handle SQL performance issues?

Determine customer-specific vs systemic issue; use actual execution plans; apply targeted index/query or broader product-safe fixes.

How to explain complex architecture to non-technical stakeholders?

Use business impact language, analogies, and explicit risk/cost/time outcomes.

How to approach technical debt?

Classify active vs stable debt, prioritize based on impact, and document decisions and rationale transparently.

How to ensure team knowledge sharing?

Document why-not-just-what, create layered onboarding material, use code reviews as teaching, and pair engineers on complex areas.
