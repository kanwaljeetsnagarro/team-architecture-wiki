Absolute Beginner Foundations

Start here if you are new to enterprise APIs, BBCRM, and distributed systems. This page removes jargon first, then builds technical depth in a guided sequence.

1) Big Picture in Plain Language

BBCRM is the core fundraising and constituent platform. It is powerful but built on legacy enterprise technology. Modern apps still need to consume it. The Translation Layer acts as a bridge so modern clients can use REST/JSON while BBCRM continues using SOAP/XML internally.

Think of this as an adapter plug: modern device on one side, legacy socket on the other.

2) Internet and API Basics You Must Know

ConceptMeaningWhy you care
ClientApp making a requestCould be web app, mobile app, or partner system
ServerApp handling requestRuns business logic and data access
EndpointSpecific URL + methodDefines what operation you are calling
PayloadData in request/response bodyJSON for REST, XML for SOAP

HTTP methods

  • GET: read data
  • POST: create data
  • PUT: replace full data
  • PATCH: partially update data
  • DELETE: remove data

Status codes

CodeMeaningAction
200/201SuccessContinue flow
400Bad requestFix request payload/validation
401UnauthenticatedAcquire or refresh token
403ForbiddenUser authenticated but no permission
404Not foundValidate identifier / route
429Too many requestsBack off / retry after window
500/503Server/downstream failureUse retry and observe logs

3) Security Basics Without Confusion

TermSimple definition
AuthenticationProve who you are (token/SSO login).
AuthorizationCheck what actions you can do.
Identity mappingMap SSO identity to BBCRM application user and context.
Common gotcha: SSO success does not guarantee BBCRM access if application-user mapping is missing.

4) End-to-End Request Journey (Beginner View)

Client -> API Management Gateway -> Translation Layer
       -> Legacy BBCRM AppFx Web Service -> Infinity Runtime -> SQL Stored Procedure
       -> Response returns back through same chain
  • Gateway handles token validation, rate limiting, routing.
  • Translation layer maps JSON fields into SOAP form structure.
  • Legacy layer executes business rules and stored procedures.
  • Response is normalized to client-friendly JSON.

6) Practical Labs for New Joiners

Lab A - Diagnose an API failure

  1. Capture status code and response error body.
  2. Find correlation ID from response headers/log context.
  3. Trace each layer in order: gateway -> translation -> downstream.
  4. Determine if issue is auth, mapping, permission, or legacy rule failure.

Lab B - Explain architecture to non-technical stakeholder

  • Use business outcome language (speed, risk, compatibility).
  • Avoid protocol jargon until asked.
  • Use one analogy and one metric-based benefit.

7) Beginner Q&A

Why do we still use legacy internals if modern APIs exist?

Because legacy internals contain proven business logic. The translation layer modernizes access without high-risk rewrite.

What is the fastest way to become useful in this team?

Master request tracing with correlation ID, understand domain vocabulary, and learn endpoint-to-operation mapping.

What should I memorize first?

HTTP methods, status codes, authentication vs authorization, and the 5-layer request flow.

7b) Live Microsoft Learn Path for Beginners

  1. API Management policies - Start with rate-limit, validate-jwt, and set-backend-service.
  2. gRPC on .NET overview - Understand where gRPC fits vs REST.
  3. Circuit Breaker pattern - Learn fail-fast and recovery logic.
  4. Compensating Transaction pattern - Learn rollback alternatives for distributed systems.
  5. Strangler Fig pattern - Learn low-risk modernization sequencing.
Recommended learning order: APIM basics -> request/response transformation -> resilience patterns -> modernization strategy.
